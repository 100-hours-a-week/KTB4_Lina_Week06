# 커뮤니티 REST API 피드백 수정 회고록

## 1. 탈퇴한 작성자 마스킹 처리 - 반영

`PostService`의 `getDetailPost`에서 작성자를 조회할 때 `.get()`을 바로 호출하고 있었다. 탈퇴한 유저는 DB에서 삭제되기 때문에 `Optional`이 비어있는 상태인데 `.get()`을 호출하면 `NoSuchElementException`이 터진다.

이 피드백을 반영해 소프트 딜리트 방식으로 마스킹 처리를 구현했다. 현업에서도 자주 사용하고, 데이터 분석 측면에서도 실제 DB에서 데이터를 날리지 않는 편이 유리하다고 생각해 소프트 딜리트를 선택했다. `User`에 `isDeleted` 필드를 추가하고, `UserRepository.delete()`가 실제로 행을 삭제하는 대신 `isDeleted = true`로만 바꾸도록 수정했다.

마스킹 처리는 `AuthorInfo`에 `from(User user)` 정적 팩토리 메서드를 추가해서 DTO 레이어에서 담당하도록 했다. `user == null`이거나 `user.isDeleted() == true`면 닉네임을 "알수없음"으로, 프로필 이미지를 null로 채워서 반환한다. `PostService`에서는 `.get()` 대신 `.orElse(null)`로 받아서 `AuthorInfo.from(author)`에 넘기는 것으로 정리됐다.

## 2. getPosts 페이지네이션 경계값 예외 - 반영

`fromIndex = (page - 1) * limit`으로 잡고 `posts.subList(fromIndex, toIndex)`를 바로 호출하고 있었다. page가 마지막 페이지를 초과하면 `fromIndex >= total`이 되어 `subList`가 `IndexOutOfBoundsException`을 던지고 500이 난다. page가 0이나 음수여도 `fromIndex`가 음수가 되어 같은 문제가 생긴다.

이 피드백을 반영해 `subList` 호출 전에 `fromIndex < 0 || fromIndex >= total` 조건을 먼저 체크해서 해당하면 빈 리스트를 바로 반환하도록 수정했다. `toIndex` 계산도 유효한 범위에서만 일어나도록 `else` 블록 안으로 옮겼다.

## 3. updateProfile 닉네임 중복 검사에서 본인 제외 - 미반영

닉네임 중복 검사가 본인까지 포함해서, 닉네임은 그대로 두고 프로필 이미지만 바꾸려 해도 `findByNickname`에 자기 자신이 걸려 `duplicate_nickname`으로 막힌다는 피드백을 받았다. 찾은 유저의 `userId`가 요청한 `userId`와 다를 때만 중복으로 보도록 조건을 한 겹 더 두면 된다는 방향은 이해했는데, 실제로 어떻게 수정해야 할지 모르겠어서 반영하지 못했다.

## 4. 게시글/회원 삭제 시 연관 데이터 처리 정책 - 반영

게시글 삭제 시 `posts`에서만 지우고 해당 글의 `comments`와 `likes`는 그대로 남아있는 문제였다. 회원 탈퇴도 `users`에서만 지우고 게시글, 댓글, 좋아요, `TokenStore`의 토큰이 남는 구조였다.

이 피드백을 반영해 게시글 삭제와 회원 탈퇴의 정책을 각각 정했다. 회원 탈퇴는 1번에서 소프트 딜리트로 처리했기 때문에 연관 데이터는 그대로 두고 작성자 정보만 마스킹하는 방향으로 정리됐다. `TokenStore`의 토큰은 탈퇴 시점에 제거한다.

게시글 삭제는 하드 딜리트로 진행했다. 삭제된 게시글의 댓글과 좋아요는 더 이상 의미가 없고, 이걸 소프트 딜리트로 관리하면 구조가 불필요하게 복잡해진다고 판단했다. `CommentRepository`와 `LikeRepository`에 `deleteByPostId()` 메서드가 없어서 각각 추가하고, `PostService.deletePost()`에서 게시글 삭제 후 댓글과 좋아요를 연달아 삭제하도록 했다.

## 5. commentsCount, viewsCount 갱신 누락 - 반영

`commentsCount`와 `viewsCount`가 초기값인 0에서 한 번도 바뀌지 않고 있었다. `likesCount`는 `pushLike()`에서 좋아요 추가/취소 시 직접 갱신하고 있었는데 댓글 카운트와 조회수는 그 코드가 빠져있었다.

이 피드백을 반영해 `likesCount`와 같은 방식으로 수정했다. `CommentService.createComment()`에서 댓글 저장 후 `commentsCount + 1`, `deleteComment()`에서 삭제 후 `commentsCount - 1`을 하고 `postRepository.update()`를 호출한다. 음수 방지를 위해 `Math.max(0, ...)` 처리도 추가했다. `viewsCount`는 `PostService.getDetailPost()` 조회 시마다 `+1` 후 업데이트하도록 했다.

## 6. getPosts 목록 응답에 author 객체 누락 - 보류

`PostListResponse`에 `Post` 엔티티를 그대로 담아서, API 명세서에는 `author` 객체로 프로필 이미지까지 준다고 했는데 실제로는 `authorId`라는 Long 값만 나가는 문제다. 목록용 응답 DTO를 따로 만들어 `author`를 객체로 담아야 한다는 방향은 이해했는데 일단 보류했다.

## 7. GlobalExceptionHandler 입력 오류 예외 처리 - 반영

`GlobalExceptionHandler`가 커스텀 예외(`BadRequestException`, `ForbiddenException`, `DuplicateException`)와 `Exception`만 잡고 있어서, 잘못된 JSON 바디나 타입 불일치 같은 입력 오류가 전부 마지막 `Exception` 핸들러에서 500으로 수렴하고 있었다.

이 피드백을 반영해 `HttpMessageNotReadableException`(JSON 파싱 실패)과 `MethodArgumentTypeMismatchException`(숫자 자리에 문자 등 타입 변환 실패) 핸들러를 `GlobalExceptionHandler`에 추가해서 각각 400으로 내려가도록 보강했다.
